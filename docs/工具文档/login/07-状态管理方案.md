# 状态管理方案（当前实现与可升级策略）

## 学习目标
- 理解项目当前使用的状态方案（React Hooks + Context）。
- 了解可替代/升级方案（Zustand、Redux、Jotai），选择合适时机引入。

## 当前方案
- 全局会话状态：由 NextAuth 的 `SessionProvider` 提供，组件通过 `useSession()` 访问。
- 本地 UI 状态：各组件通过 `useState` 管理（表单、loading/error、一次性提示）。
- 服务层：`authService.ts` 封装登录/退出逻辑，减少组件内状态耦合。

## 方案优缺点
- 优点：零额外依赖、易上手、足够满足中小型应用。
- 缺点：数据需要层层传递时可能样板代码增多；跨页面共享非会话的复杂状态不够优雅。

## 可升级方案（何时考虑）
- Zustand：简单、轻量、无样板，适合共享少量 UI/业务状态。
- Redux：可预测数据流、开发者工具强大；适合大型复杂应用。
- Jotai/Recoil：原子化状态管理，易于拆分与组合。

## 本项目建议
- 继续沿用 Context + Hooks 管理会话与局部状态。
- 当出现跨页面复杂 UI 状态共享（如通知中心、全局过滤器）时再评估引入 Zustand。

## 练习建议
- 将一次性提示提取为 `useOneTimeBanner()` Hook，并在多个页面复用。
- 构建一个轻量 `useAuthDialog()` 管理登录弹窗开关与结果。
